    1. Import `WorkflowRunEvent` from `@octokit/webhooks-types`
    2. Import `{ buildMetricTags, emitTriggerCount, emitOutcome, emitRunDuration, emitCostEstimate, emitCircuitBreakerTrip, emitScopeViolation }` from `../metrics.js`
    3. Import `{ estimateCostUsd }` from `../metrics.js` and a helper to extract run duration
    4. Change `payload: any` to `payload: WorkflowRunEvent`
    5. Build tags: `const tags = buildMetricTags(payload.repository.full_name)`
    6. Always emit: `emitTriggerCount(tags)` (OPS-01)
    7. Extract run duration from `payload.workflow_run.run_started_at` and `payload.workflow_run.updated_at` -> ms delta. Emit `emitRunDuration(durationMs, tags)` (OPS-04)
    8. Estimate cost: `const cost = estimateCostUsd(durationMs)` -> `emitCostEstimate(cost, tags)` (VAL-03, VAL-04, SAFE-01)
    9. Classify outcome using the 5-way FixOutcome:
       - `conclusion === 'success'` -> `fix_pr_created`
       - `conclusion === 'failure'` with workflow name containing 'auto-fix' -> check head_branch or name patterns:
         - If name/logs suggest circuit breaker -> `circuit_breaker`, also `emitCircuitBreakerTrip(tags)` (SAFE-02)
         - If name/logs suggest scope violation -> `no_fix`, also `emitScopeViolation(tags)` (SAFE-03)
         - If name suggests flaky skip -> `flaky_skipped`
         - Default failure -> `no_fix`
       - `conclusion === 'cancelled'` -> `escalated`
       **Note:** Since we cannot query workflow logs from webhooks, use a practical heuristic: map conclusion directly. `success` = `fix_pr_created`, `failure` = `no_fix`, `cancelled` = `escalated`. The circuit_breaker and flaky_skipped outcomes can be refined later when naming conventions are established in the caller workflows. For now, emit the `no_fix` outcome for failures and add a breadcrumb noting the conclusion for future refinement.
    10. Emit `emitOutcome(outcome, tags)` (OPS-02)
    11. Keep existing Sentry breadcrumb, enhance data with outcome classification

    **pull-request handler (api/lib/handlers/pull-request.ts):**
    1. Import `PullRequestEvent` from `@octokit/webhooks-types`
    2. Import `{ buildMetricTags, emitPrAccepted, emitPrRejected, emitMTTR, computeMttrMs }` from `../metrics.js`
    3. Change `payload: any` to `payload: PullRequestEvent`
    4. Build tags: `const tags = buildMetricTags(payload.repository.full_name)`
    5. On `action === 'closed'`:
       - If `payload.pull_request.merged === true`: `emitPrAccepted(tags)` (VAL-02)
       - If merged is false: `emitPrRejected(tags)` (VAL-02) -- also `emitScopeViolation(tags)` for closed-without-merge as potential scope violation (SAFE-03)
    6. MTTR calculation (VAL-01): On merged PRs, attempt MTTR. The PR `created_at` is available. For the workflow failure timestamp, we do NOT have it in this payload. Store a Sentry breadcrumb noting MTTR requires correlation with workflow_run data. For now, skip MTTR emission in this handler -- MTTR will be a future enhancement when we add cross-event correlation. Add a `// TODO(v1.3): MTTR requires workflow_run.run_started_at which is not in the PR payload` comment.
       **Alternative for VAL-01:** Emit MTTR from the workflow_run handler instead when conclusion is success -- the workflow_run payload HAS both `run_started_at` (failure time proxy) and `updated_at` (completion time). The PR creation time can be approximated as ~equal to workflow completion for auto-fix. So in workflow-run handler, when outcome is `fix_pr_created`, emit `emitMTTR(durationMs, tags)` using the run duration as MTTR proxy (time from CI start to fix completion). This is the pragmatic approach per user decision "MTTR computed from webhook payload timestamps only."
    7. Keep existing Sentry breadcrumb
