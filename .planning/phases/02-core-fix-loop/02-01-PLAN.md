Key architecture from Phase 1:
- `auto-fix.yml` is the central reusable workflow with `workflow_call` trigger
- Central repo checked out to `_auto-fix-scripts/` path; failing repo is main working directory
- Circuit breaker (step 4) checks branch name and PR labels -- output: `steps.circuit.outputs.should_proceed`
- Log retrieval (step 5) uses env indirection and `sanitize-logs.sh`
- Prompt loading (step 6) reads from `_auto-fix-scripts/config/repo-stack-map.json`
- Claude Code Action (step 7) uses `--max-turns 10` and restricted `--allowedTools`

Key constraints from CONTEXT.md / RESEARCH.md:
- `workflow_run` requires explicit workflow names (no wildcard) -- each caller must list CI workflow names
- `workflow_run` event fires on ALL branches by default (user decision)
- Caller workflow must stay under 15 lines (CIFD-04)
- Re-run mechanism: `github.rest.actions.reRunWorkflowFailedJobs()` mutates the existing run (same run_id)
- Poll with `getWorkflowRun` on same run_id until status == 'completed'
- If re-run passes: add annotation via `core.notice()` and skip agent
- 30-minute max wait with 15-second polling intervals
- `GITHUB_REF` in workflow_run context is the DEFAULT branch, NOT the failed run's branch -- must pass `head_branch` explicitly

**3. Add flakiness filter step (new step between circuit breaker and log retrieval):**

Insert a new step AFTER the circuit breaker (step 4) and BEFORE the "Get and sanitize failed logs" step (currently step 5). The flakiness filter uses `actions/github-script@v7`:

```yaml
# 5. Flakiness filter (CIFD-03)
# Re-runs failed jobs once. If the re-run passes, skips the agent.
- name: Flakiness filter
  id: flaky
  if: steps.circuit.outputs.should_proceed == 'true'
  uses: actions/github-script@v7
  with:
    github-token: ${{ steps.app-token.outputs.token }}
    script: |
      const [owner, repoName] = '${{ inputs.repository }}'.split('/');
      const runId = parseInt('${{ inputs.failed_run_id }}');

      // Re-run only failed jobs (mutates the existing run -- same run_id)
      await github.rest.actions.reRunWorkflowFailedJobs({
        owner, repo: repoName, run_id: runId
      });
      core.info(`Re-running failed jobs for run ${runId}`);

      // Poll until completion (15s intervals, 30min safety cap)
      const maxWait = 30 * 60 * 1000;
      const start = Date.now();
      let conclusion = null;
      while (!conclusion && (Date.now() - start) < maxWait) {
        await new Promise(r => setTimeout(r, 15000));
        const { data: run } = await github.rest.actions.getWorkflowRun({
          owner, repo: repoName, run_id: runId
        });
        if (run.status === 'completed') {
          conclusion = run.conclusion;
        }
      }

      if (conclusion === 'success') {
        core.notice('Flaky failure detected -- passed on re-run, agent skipped');
        core.setOutput('is_flaky', 'true');
      } else {
        core.info(`Re-run concluded: ${conclusion ?? 'timeout'}`);
        core.setOutput('is_flaky', 'false');
      }
```

```yaml
# Auto-Fix Caller — copy to your repo's .github/workflows/auto-fix-caller.yml
# Customize the 'workflows' list to match your CI workflow name(s).
# The workflow name must match the 'name:' field in your CI workflow YAML exactly.
name: Auto Fix
on:
  workflow_run:
    workflows: [CI]  # CHANGE THIS: list your CI workflow name(s), e.g. [CI, Build, Test]
    types: [completed]
jobs:
  fix:
    if: github.event.workflow_run.conclusion == 'failure'
    uses: fbetancourtc/auto-fix-agent/.github/workflows/auto-fix.yml@main
    with:
      app_id: "2985828"
      failed_run_id: "${{ github.event.workflow_run.id }}"
      repository: "${{ github.repository }}"
      head_branch: "${{ github.event.workflow_run.head_branch }}"
    secrets:
      anthropic_api_key: "${{ secrets.ANTHROPIC_API_KEY }}"
      app_private_key: "${{ secrets.AUTO_FIX_APP_PRIVATE_KEY }}"
```

Key design notes:
- The App ID is hardcoded to `2985828` (from Phase 1 summary) because it's a constant -- every caller uses the same GitHub App
- `workflow_run.workflows` requires exact `name:` matches -- the comment instructs users to customize
- `head_branch` passes `github.event.workflow_run.head_branch` for correct branch checkout
- `github.event.workflow_run.id` provides the failed run ID
- `github.repository` provides `owner/repo` format
- Secrets are passed explicitly (NOT `secrets: inherit` -- fails silently cross-org)
- The `if` condition filters to failures only
- The file is ~13 YAML lines (well under the 15-line limit)

IMPORTANT: This file goes in the CENTRAL repo as an example/template. Each pilot repo copies it and customizes the `workflows` list to match their CI workflow names. The caller workflow file must exist on the repo's default branch (usually `main` or `develop`) for `workflow_run` to trigger -- document this in a comment at the top.
