**Add a new job `handle-decline`** after the existing `promote` job:

```yaml
  handle-decline:
    # Close qa→main PR and add needs-human label when reviewer requests changes
    if: >-
      github.event.review.state == 'changes_requested' &&
      contains(github.event.pull_request.labels.*.name, 'auto-fix-promote') &&
      github.event.pull_request.base.ref == 'main'
    runs-on: ubuntu-latest
    timeout-minutes: 2
    permissions:
      pull-requests: write
      issues: write
    steps:
      - name: Close PR and add needs-human label
        env:
          GH_TOKEN: ${{ github.token }}
          PR_NUMBER: ${{ github.event.pull_request.number }}
          REPO: ${{ github.repository }}
        run: |
          echo "Reviewer requested changes on qa→main PR #$PR_NUMBER"
          gh pr edit "$PR_NUMBER" --repo "$REPO" --add-label "needs-human"
          gh pr close "$PR_NUMBER" --repo "$REPO" --comment "Closed by auto-fix promotion system: reviewer requested changes. Label 'needs-human' added. The fix remains on develop/qa but will not be promoted to main automatically."
          echo "PR #$PR_NUMBER closed with needs-human label"
```

**Key design decisions:**
- Uses `github.token` (not App token) since the workflow runs in the target repo where the PR lives -- no cross-org auth needed for this action.
- Condition checks THREE things: (1) review state is `changes_requested`, (2) PR has the `auto-fix-promote` label (only fires on promotion PRs, not normal PRs), (3) base ref is `main` (only fires on qa→main PRs, not develop→qa PRs).
- `timeout-minutes: 2` since the operation is trivial.
- `permissions: pull-requests: write, issues: write` -- needs write to close PR and add labels.
- Adds both a label AND a closing comment explaining what happened.

**Preserve the existing `promote` job exactly as-is.** Only add the new trigger event and new job.

The final file should have:
- `on:` with both `pull_request: types: [closed]` and `pull_request_review: types: [submitted]`
- `jobs.promote:` unchanged (existing promotion logic)
- `jobs.handle-decline:` new job for reviewer decline handling

**Important:** The `promote` job's `if:` condition already gates on `github.event.pull_request.merged == true` which will be false for `pull_request_review` events (merged is not set), so the promote job will naturally skip on review events. The `handle-decline` job's `if:` condition gates on `github.event.review.state` which is only set on `pull_request_review` events, so it will naturally skip on `pull_request.closed` events.
