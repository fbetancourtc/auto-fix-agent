**Step 1: Audit each repo's CI workflow names and directory structure.**

For each of the 8 repos, use the GitHub CLI to discover CI workflow names and check directory structure:

```bash
# For each repo, extract CI workflow name(s)
gh api repos/{owner}/{repo}/contents/.github/workflows --jq '.[].name' 2>/dev/null
# Then for each workflow file:
gh api repos/{owner}/{repo}/contents/.github/workflows/{filename} --jq '.content' | base64 -d | grep "^name:"
```

Also check each repo's default branch:
```bash
gh repo view {owner}/{repo} --json defaultBranchRef --jq '.defaultBranchRef.name'
```

For Python repos (binance-bot, conciliacion-averias), briefly audit directory structure to check if `src/`, `app/`, `lib/`, `tests/` defaults are appropriate, or if custom `allowed_dirs` are needed in repo-stack-map.json.

Record findings in a structured list:
- Repo name
- CI workflow name(s) (exact `name:` field)
- Default branch (main or develop)
- Needs custom allowed_dirs? (yes/no + directories if yes)

**Step 2: Update repo-stack-map.json if any repo needs custom allowed_dirs.**

If any repo has source code outside the stack default directories, add a per-repo `allowed_dirs` override. Most repos should be fine with stack defaults.

**Step 3: Deploy caller workflow to each repo.**

For each of the 8 repos, create the caller workflow file using the GitHub API. The caller is based on `auto-fix-caller.example.yml` but with the `workflows:` list customized to match that repo's CI workflow name(s).

For each repo:
```bash
# Create the file content with the correct workflows list
# Use gh api to create the file on the repo's default branch
gh api repos/{owner}/{repo}/contents/.github/workflows/auto-fix-caller.yml \
  --method PUT \
  --field message="ci: add auto-fix caller workflow" \
  --field content="$(base64 <<< 'CALLER_CONTENT_HERE')" \
  --field branch="{default_branch}"
```

CRITICAL: The caller MUST be on the default branch (not a PR) for `workflow_run` to trigger. Use direct file creation via API, not PRs. Per user decision, Claude picks delivery method -- for fbetancourtc (full admin), direct API push is appropriate. For LiftitFinOps, use the same approach if admin access allows, otherwise create a PR and document the merge requirement.

Each caller file is identical to the template except:
- `workflows:` list matches the repo's actual CI workflow name(s)

Do NOT modify the reusable workflow reference, app_id, or secrets block.
