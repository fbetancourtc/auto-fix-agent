<decisions>
## Implementation Decisions

### Evidence Standard
- Foundation requirements (FOUND-01–04): verify artifact existence and correct structure (file present, config correct, code structure matches requirement description)
- Security requirements (SECR-01–04): stricter standard — read the actual implementation logic (sanitization code, circuit breaker logic, token limit enforcement, scope restriction) and confirm correctness
- Re-verify all 8 requirements from scratch — do not trust existing REQUIREMENTS.md checkmarks
- External state that can't be verified from code alone (e.g., FOUND-02 GitHub App installed on all 3 orgs, FOUND-03 repo is public): mark as "cannot verify locally" with notes on what to manually confirm

### Gap Handling
- Document gaps and proceed — VERIFICATION.md lists gaps with details, milestone audit decides whether they block completion
- Known limitations with external dependencies (e.g., `allowedTools` enforcement gap from claude-code-action issue #860): acknowledge as "known gap — external dependency" and don't count against the requirement if a workaround exists in the codebase
- VERIFICATION.md includes per-requirement status (satisfied / gap / known-limitation) plus an overall "X of 8 satisfied" summary with pass/fail verdict
- Verifier updates REQUIREMENTS.md checkboxes based on evidence found — keep single source of truth current

### Verification Scope
- Include Phase 2.1 fixes in scope — they patched Phase 1 artifacts (validate-diff.sh, circuit breaker, test-guardrails, README, Python prompt sync, label docs), so verifying Phase 1 code naturally covers them
- Use Phase 1 plan summaries (01-01-SUMMARY.md, 01-02-SUMMARY.md) as reference for what was intended, then verify actual code matches
- Check both requirement IDs AND Phase 1 success criteria from ROADMAP.md — requirements confirm features exist, success criteria confirm they work together as intended
- Skip Phase 1 research (01-RESEARCH.md) as reference — plans and requirements are the authoritative sources

### Claude's Discretion
- VERIFICATION.md internal structure and formatting
- Order of requirement verification
- How to present evidence for each requirement (code snippets, file references, etc.)
- Whether to group findings by requirement category or verify in ID order

</decisions>
