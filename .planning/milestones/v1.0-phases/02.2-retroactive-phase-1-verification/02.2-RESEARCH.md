## Summary

The single deliverable is `01-VERIFICATION.md` at `.planning/phases/01-infrastructure-and-security-guardrails/01-VERIFICATION.md`, plus updated checkboxes in `.planning/REQUIREMENTS.md`.

### Anti-Patterns to Avoid

- **Trusting REQUIREMENTS.md checkmarks:** CONTEXT.md decision is to re-verify from scratch. Ignore existing `[x]` marks during verification.
- **Marking FOUND-02/03 as GAP:** These are external state requirements, not code gaps. Use CANNOT VERIFY LOCALLY.
- **Conflating `allowedTools` enforcement gap with SECR-04 failure:** The `allowedTools` gap (issue #860) concerns client-side enforcement of tool restrictions inside Claude Code Action. SECR-04 concerns secrets exposure in the env block. The diff validation script provides the primary enforcement mechanism for scope restriction.
- **Skipping Phase 2.1 fixes:** The CONTEXT.md decision explicitly includes Phase 2.1 fixes in scope. The fail-open circuit breaker behavior (SECR-02) and the SECR-04 GH_TOKEN allowance are Phase 2.1 contributions to Phase 1 requirements.

## Common Pitfalls

### Pitfall 1: Marking REQUIREMENTS.md Incorrectly After Verification

**What goes wrong:** Verifier updates all 8 checkboxes to `[x]` without distinguishing between "satisfied" and "cannot verify locally."
**Why it happens:** CANNOT VERIFY LOCALLY requirements are not gaps — they are satisfied insofar as the code is concerned, with external confirmation pending.
**How to avoid:** Update REQUIREMENTS.md checkboxes: use `[x]` for SATISFIED and CANNOT VERIFY LOCALLY (the requirement is met at the code level), use `[ ]` only for actual GAPs.
**Warning signs:** If the verifier removes a `[x]` for FOUND-02 or FOUND-03, reconsider.

### Pitfall 2: Conflating SECR-04 Requirement with allowedTools Issue #860

**What goes wrong:** Verifier marks SECR-04 as a gap because `allowedTools` enforcement is not fully guaranteed by the client.
**Why it happens:** The requirement is "agent never has access to production secrets or deployment triggers" — which is about secrets in the env block, NOT about tool restriction enforcement.
**How to avoid:** SECR-04 is satisfied by: (a) no production secrets in agent step env, (b) minimal workflow permissions, (c) diff validation as post-hoc enforcement. The `allowedTools` issue is a known limitation for scope restriction, not a SECR-04 failure.
**Warning signs:** If SECR-04 evidence refers to allowedTools enforcement rather than secrets exposure, the check is addressing the wrong thing.

### Pitfall 3: Missing the Phase 2.1 Contributions

**What goes wrong:** Verifier evaluates the circuit breaker without the try/catch (pre-Phase 2.1 state) and marks SECR-02 as a gap.
**Why it happens:** Reading Phase 1 plan summaries without cross-checking the current codebase state.
**How to avoid:** Always verify against the current codebase files, not against Phase 1 plan summaries. Phase 2.1 is explicitly in scope per CONTEXT.md.
**Warning signs:** If SECR-02 evidence doesn't mention `try/catch` or `core.warning`, the verifier is reading an outdated state.

### Pitfall 4: Wrong Output Path

**What goes wrong:** VERIFICATION.md written to the phase 2.2 directory instead of the phase 1 directory.
**Why it happens:** Confusion between "phase being executed" and "phase being verified."
**How to avoid:** Output path is `.planning/phases/01-infrastructure-and-security-guardrails/01-VERIFICATION.md`. The phase 2.2 directory only holds planning artifacts (CONTEXT.md, RESEARCH.md, PLAN.md).

### SECR-01: Sanitize-Logs.sh Structure (5 Passes)

```bash
# Source: scripts/sanitize-logs.sh
# Pass 1: Truncate to 500 lines
tail -500 "$INPUT_FILE" > "$OUTPUT_FILE"

# Pass 2: Strip prompt injection patterns (case-insensitive)
sed -i.bak -E \
  -e '/-- [Aa]dditional.*[Ii]nstruction/d' \
  -e '/IMPORTANT: [Yy]ou must/d' \
  -e '/^[Yy]ou are a /d' \
  -e '/[Ii]gnore previous.*[Ii]nstructions/d' \
  -e '/[Dd]isregard.*[Ii]nstructions/d' \
  -e '/<\/?[Ss][Yy][Ss][Tt][Ee][Mm]>/d' \
  -e '/\[\/?\/?[Ss][Yy][Ss][Tt][Ee][Mm]\]/d' \
  "$OUTPUT_FILE"

# Pass 3: Strip shell injection (backtick and $())
sed -i.bak -E \
  -e 's/`[^`]+`/[REMOVED]/g' \
  -e 's/\$\([^)]+\)/[REMOVED]/g' \
  "$OUTPUT_FILE"

# Pass 4: Redact GitHub tokens (ghp_, ghs_, github_pat_, sk-)
sed -i.bak -E \
  -e 's/ghp_[A-Za-z0-9]{36,}/[REDACTED]/g' \
  -e 's/ghs_[A-Za-z0-9]{36,}/[REDACTED]/g' \
  -e 's/github_pat_[A-Za-z0-9_]{20,}/[REDACTED]/g' \
  -e 's/sk-[A-Za-z0-9_-]{20,}/[REDACTED]/g' \
  "$OUTPUT_FILE"

# Pass 5: Redact secret assignments
sed -i.bak -E \
  -e 's/(ANTHROPIC_API_KEY=).*/\1[REDACTED]/' \
  -e 's/(AWS_SECRET_ACCESS_KEY=).*/\1[REDACTED]/' \
  -e 's/(GITHUB_TOKEN=).*/\1[REDACTED]/' \
  "$OUTPUT_FILE"
```

**Test coverage in test-guardrails.yml:** Lines 26–86 test sanitization with known injection payloads — confirms all injection categories are stripped and `[REDACTED]` markers are produced while normal log content is preserved.

### SECR-02: Circuit Breaker with Fail-Open (Phase 2.1)

```javascript
// Source: .github/workflows/auto-fix.yml, lines 82–124
// Check 1: Fail open on getWorkflowRun API error
let headBranch;
try {
  const run = await github.rest.actions.getWorkflowRun({
    owner, repo: repoName, run_id: runId
  });
  headBranch = run.data.head_branch;
} catch (e) {
  core.warning(`Circuit breaker: getWorkflowRun failed (failing open): ${e.message}`);
  core.setOutput('should_proceed', 'true');
  return;
}

// Check 2: Block auto-fix branches
if (headBranch && headBranch.startsWith('auto-fix/')) {
  core.setOutput('should_proceed', 'false');
  return;
}

// Check 3: Block open PRs with auto-fix label (fails open on API error)
try {
  const pulls = await github.rest.pulls.list({ ... });
  for (const pr of pulls.data) {
    if (pr.labels.some(l => l.name === 'auto-fix')) {
      core.setOutput('should_proceed', 'false');
      return;
    }
  }
} catch (e) {
  core.warning(`Circuit breaker: could not check PRs: ${e.message}`);
}

core.setOutput('should_proceed', 'true');
```
